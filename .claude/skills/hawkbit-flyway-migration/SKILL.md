# Hawkbit Flyway Migration

Hawkbit ships three parallel SQL migration files for every schema change — one per supported database engine. This skill ensures all three files are correct, consistent, and follow project conventions.

## Where Migrations Live

All migration SQL files go in one module:

```
hawkbit-repository/hawkbit-repository-jpa-flyway/
  src/main/resources/db/migration/
    H2/
    MYSQL/
    POSTGRESQL/
```

Each directory contains the same logical migration, written in that engine's dialect. Flyway selects the right folder at runtime via `spring.flyway.sql-migration-suffixes=${spring.jpa.database}.sql`.

## File Naming

```
V<major>_<minor>_<patch>__<description>__<DATABASE>.sql
```

**Pattern breakdown:**
- `V` — versioned migration prefix (Flyway convention)
- `<major>_<minor>_<patch>` — version number using underscores (Flyway treats them as dots: `1_20_1` → `1.20.1`)
- `__` — double-underscore separator (required by Flyway)
- `<description>` — snake_case summary of the change
- `__<DATABASE>` — engine suffix: `H2`, `MYSQL`, or `POSTGRESQL`
- `.sql` — extension

**Determining the next version:** Check what already exists:

```bash
ls hawkbit-repository/hawkbit-repository-jpa-flyway/src/main/resources/db/migration/H2/V*.sql | sort
```

Increment the patch number from the highest existing version. The current baseline is `1_20_0` and the latest incremental is `V1_20_1`, so the next migration would be `V1_20_2__your_description__*.sql`.

**Example set for a new migration:**
```
V1_20_2__add_priority_to_action__H2.sql
V1_20_2__add_priority_to_action__MYSQL.sql
V1_20_2__add_priority_to_action__POSTGRESQL.sql
```

Write all three files for every migration. No exceptions.

## Type Mapping Across Engines

The three engines differ in specific ways. Use this reference when writing column definitions:

| Concept | H2 | MySQL/MariaDB | PostgreSQL |
|---------|-----|---------------|------------|
| Auto-increment PK | `BIGINT GENERATED BY DEFAULT AS IDENTITY` | `BIGINT NOT NULL AUTO_INCREMENT` | `BIGINT GENERATED BY DEFAULT AS IDENTITY` |
| Timestamp | `TIMESTAMP` | `DATETIME(6)` | `TIMESTAMP` |
| Large text | `CLOB` | `TEXT` | `TEXT` |
| Boolean | `BOOLEAN` | `BOOLEAN` | `BOOLEAN` |
| Integer | `INTEGER` | `INTEGER` | `INTEGER` |
| Float | `FLOAT` | `FLOAT` | `FLOAT` |
| Varchar | `VARCHAR(n)` | `VARCHAR(n)` | `VARCHAR(n)` |
| Char | `CHAR(n)` | `CHAR(n)` | `CHAR(n)` |
| Bigint | `BIGINT` | `BIGINT` | `BIGINT` |

H2 and PostgreSQL are nearly identical — the main differences are:
- `CLOB` (H2) vs `TEXT` (PostgreSQL) for large text
- PostgreSQL sometimes needs extra indexes for FK columns (see PostgreSQL-specific section)

MySQL is the most different:
- `AUTO_INCREMENT` instead of identity columns
- `DATETIME(6)` instead of `TIMESTAMP`
- `TEXT` instead of `CLOB`

## Standard Column Patterns

### Audit Columns (all tenant-aware entities)

Every entity that extends `AbstractJpaTenantAwareBaseEntity` has these columns:

```sql
id BIGINT ...,               -- PK, engine-specific identity syntax
created_at BIGINT,
created_by VARCHAR(64),
last_modified_at BIGINT,
last_modified_by VARCHAR(64),
optlock_revision BIGINT,
tenant VARCHAR(40) NOT NULL,
```

Timestamps are stored as epoch milliseconds (`BIGINT`), not as SQL timestamp types. The `optlock_revision` column supports JPA optimistic locking (`@Version`).

### Common Column Sizes

These sizes are conventions from the existing schema — follow them for consistency:

| Content | Size |
|---------|------|
| Entity names | `VARCHAR(128)` |
| Descriptions | `VARCHAR(512)` |
| Tenant ID | `VARCHAR(40)` |
| Versions | `VARCHAR(64)` |
| Type keys | `VARCHAR(64)` |
| User identifiers | `VARCHAR(64)` |
| Colours | `VARCHAR(16)` |
| Controller IDs / addresses | `VARCHAR(256)` or `VARCHAR(512)` |
| Metadata keys | `VARCHAR(128)` |
| Metadata values | `VARCHAR(4000)` |
| SHA-256 hashes | `CHAR(64)` |
| SHA-1 hashes | `VARCHAR(40)` |
| MD5 hashes | `VARCHAR(32)` |

### Column Naming

JPA maps camelCase Java fields to snake_case SQL columns. Follow the existing conventions:

| Java field | SQL column |
|------------|-----------|
| `controllerId` | `controller_id` |
| `installationDate` | `install_date` |
| `assignedDistributionSet` | `assigned_distribution_set` |
| `distributionSet` | `distribution_set` |
| FK to type table | `ds_type`, `sm_type`, `target_type` |
| FK to parent | `rollout`, `action`, `target`, `software_module` |

Note: FK column names are often short references (e.g., `ds` not `distribution_set_id`, `sm` not `software_module_id`) — check the existing schema for the specific entity's convention.

## Index Conventions

### Primary Tenant Index

Every tenant-aware table gets a primary lookup index with `tenant` as the leading column:

```sql
CREATE INDEX sp_idx_<table>_prim ON <table> (tenant, id);
```

This is critical for multi-tenant query performance since almost every query filters by tenant.

### Unique Indexes

Unique business constraints always include `tenant`:

```sql
CREATE UNIQUE INDEX uk_<table> ON <table> (<business_key_columns>, tenant);
-- or
CREATE UNIQUE INDEX uk_<table>_<suffix> ON <table> (<columns>, tenant);
```

Examples:
```sql
CREATE UNIQUE INDEX uk_target_controller_id ON sp_target (controller_id, tenant);
CREATE UNIQUE INDEX uk_rollout ON sp_rollout (name, tenant);
CREATE UNIQUE INDEX uk_distribution_set ON sp_distribution_set (tenant, name, version, ds_type);
```

### Regular Indexes

Numbered sequentially per table:

```sql
CREATE INDEX sp_idx_<table>_01 ON <table> (tenant, <columns>);
CREATE INDEX sp_idx_<table>_02 ON <table> (tenant, <columns>);
```

`tenant` is the leading column in most indexes because the discriminator filter narrows the scan range.

### Index Naming Summary

| Type | Pattern | Example |
|------|---------|---------|
| Primary tenant index | `sp_idx_<table>_prim` | `sp_idx_target_prim` |
| Numbered query index | `sp_idx_<table>_NN` | `sp_idx_target_01` |
| Unique constraint | `uk_<table>` or `uk_<table>_<suffix>` | `uk_rollout`, `uk_target_controller_id` |
| FK support index | `fk_<constraint_name>` or `sp_idx_<table>_<ref>` | `sp_idx_action_target` |

## Foreign Key Conventions

### Naming

```sql
ALTER TABLE <child_table>
    ADD CONSTRAINT fk_<child_short>_<column_or_ref> FOREIGN KEY (<column>) REFERENCES <parent_table> (id);
```

Examples:
```sql
fk_action_distribution_set    -- sp_action.distribution_set → sp_distribution_set.id
fk_target_relation_target_type -- sp_target.target_type → sp_target_type.id
fk_artifact_software_module   -- sp_artifact.software_module → sp_software_module.id
```

### CASCADE Rules

Pick the right cascade action based on the relationship:

| Relationship | ON DELETE | When |
|-------------|-----------|------|
| Strong ownership (child can't exist without parent) | `CASCADE` | Join tables, metadata, action_status, tags |
| Optional reference | `SET NULL` | `target_type` on target, auto_assign_ds on filter |
| Critical reference (prevent orphans) | *(omit — defaults to RESTRICT)* | distribution_set on action, rollout on action |

## PostgreSQL-Specific Differences

PostgreSQL occasionally needs extra indexes that H2 and MySQL don't. Check the baseline files — PostgreSQL has these extras:

```sql
-- PostgreSQL adds explicit FK support indexes
CREATE INDEX sp_idx_target_attributes_target_id ON sp_target_attributes (target);
CREATE INDEX sp_idx_action_target ON sp_action (target);
CREATE INDEX sp_idx_rollout_target_group_target ON sp_rollout_target_group (target);
```

When adding a foreign key in PostgreSQL, consider whether the FK column needs its own index for efficient cascading deletes or joins. If the table is a join table or frequently queried by that FK column, add an index in the PostgreSQL variant.

## Creating a New Table

Follow this template. Write one file per engine, adjusting identity and type syntax:

### H2 Template

```sql
CREATE TABLE sp_<entity> (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    created_at BIGINT,
    created_by VARCHAR(64),
    last_modified_at BIGINT,
    last_modified_by VARCHAR(64),
    optlock_revision BIGINT,
    tenant VARCHAR(40) NOT NULL,
    -- entity-specific columns here
    PRIMARY KEY (id)
);
CREATE UNIQUE INDEX uk_<entity> ON sp_<entity> (<business_key>, tenant);
CREATE INDEX sp_idx_<entity>_prim ON sp_<entity> (tenant, id);
-- additional indexes as needed
ALTER TABLE sp_<entity>
    ADD CONSTRAINT fk_<entity>_<ref> FOREIGN KEY (<fk_column>) REFERENCES <parent_table> (id);
```

### MySQL Template

Same structure, but change:
```sql
id BIGINT NOT NULL AUTO_INCREMENT,
```
And use `TEXT` instead of `CLOB`, `DATETIME(6)` instead of `TIMESTAMP`.

### PostgreSQL Template

Same as H2 (both use `GENERATED BY DEFAULT AS IDENTITY` and `TIMESTAMP`), but:
- Use `TEXT` instead of `CLOB`
- Add extra FK-support indexes where needed for cascade performance

## Adding a Column

For `ALTER TABLE` migrations, write one statement per file:

### H2

```sql
ALTER TABLE sp_<table> ADD COLUMN <column_name> <H2_TYPE> [NOT NULL] [DEFAULT <value>];
```

### MySQL

```sql
ALTER TABLE sp_<table> ADD COLUMN <column_name> <MYSQL_TYPE> [NOT NULL] [DEFAULT <value>];
```

### PostgreSQL

```sql
ALTER TABLE sp_<table> ADD COLUMN <column_name> <PG_TYPE> [NOT NULL] [DEFAULT <value>];
```

If adding a NOT NULL column to an existing table with data, provide a DEFAULT value so existing rows don't break. Alternatively, add as nullable, backfill, then alter to NOT NULL in a subsequent migration.

## Adding an Index

Index DDL is identical across all three engines:

```sql
CREATE INDEX sp_idx_<table>_NN ON sp_<table> (tenant, <columns>);
```

If the index DDL is the same for all engines, the three files will have identical content — that's fine.

## Safe Online-Migration Patterns

Large tenant tables (`sp_target`, `sp_action`, `sp_action_status`) can have millions of rows in production. Take care with migrations that touch them:

### Adding a Nullable Column

Safe on all engines — no rewrite needed:

```sql
ALTER TABLE sp_target ADD COLUMN priority INTEGER;
```

### Adding a NOT NULL Column with Default

```sql
-- H2 / PostgreSQL (fast — metadata-only in modern versions)
ALTER TABLE sp_target ADD COLUMN priority INTEGER DEFAULT 0 NOT NULL;

-- MySQL (may rewrite table on older versions — still safe, just slow)
ALTER TABLE sp_target ADD COLUMN priority INTEGER DEFAULT 0 NOT NULL;
```

On PostgreSQL 11+ and H2, adding a column with a DEFAULT is a metadata-only operation. On MySQL/MariaDB with InnoDB, it may rewrite the table depending on version, but it's still safe — just potentially slow on very large tables.

### Creating an Index Concurrently

Standard `CREATE INDEX` acquires a write lock. For very large tables in production:

- **PostgreSQL** supports `CREATE INDEX CONCURRENTLY` — but Flyway can't run it inside a transaction. If you need this, use a separate migration file with `-- flyway:executeInTransaction=false` at the top.
- **MySQL/MariaDB** supports `ALTER TABLE ... ADD INDEX` as an online DDL operation (InnoDB default).
- **H2** — not a concern since H2 is only used for testing/development.

### Dropping a Column

Avoid dropping columns in a migration unless you're sure no running code reads them. Prefer a two-phase approach:

1. **First release:** Deploy code that stops writing to the column
2. **Next release:** Drop the column

### Renaming a Column

Don't rename. Instead: add new column → backfill → update code → drop old column across multiple releases.

## Checklist Before Committing

Before finalizing your migration files:

1. Three files exist — one in each of `H2/`, `MYSQL/`, `POSTGRESQL/`
2. All three share the same version number and description
3. Identity column syntax matches the target engine
4. `CLOB`/`TEXT`/`DATETIME(6)` types are correct per engine
5. New tenant-aware tables include `tenant VARCHAR(40) NOT NULL`
6. `sp_idx_<table>_prim` index on `(tenant, id)` exists for new tables
7. Unique indexes include `tenant`
8. FK constraint names follow `fk_<table>_<reference>` pattern
9. PostgreSQL file has any extra FK-support indexes
10. NOT NULL columns on existing tables have DEFAULT values

## Relationship to JPA Entities

Migration files must stay in sync with JPA entity definitions in `hawkbit-repository/hawkbit-repository-jpa`. When modifying a `Jpa*` entity class:

- Adding `@Column(name = "new_col")` → write an `ALTER TABLE ADD COLUMN` migration
- Adding `@ManyToOne` with `@JoinColumn` → write the column + FK constraint + index migration
- Adding `@ElementCollection` with `@CollectionTable` → write a `CREATE TABLE` migration for the collection table
- Adding `@Table` on a new entity → write a full `CREATE TABLE` migration

The entity annotation is the source of truth for column names, sizes, and nullability. The migration must match exactly what JPA expects.
